# ADR-003: Kafka (MSK) as the Canonical Event Bus

**Status:** Accepted
**Date:** 2026-04-27
**Author:** Danny Goldstein

## Context

AEOS services need to communicate asynchronously. The data flow is:

```
Telemetry → Intelligence → Recommendations / Governance
Assessment → Process Discovery → Intelligence
```

Services must be decoupled — they deploy independently, are owned by different teams (internal + Trigent + Arkos), and have different release cadences. Direct HTTP calls between services for async workflows create tight coupling, shared failure domains, and ordering problems.

Requirements:

1. **Ordering guarantees.** Events within a UoP execution must be processed in order (span_enriched before ledger.row_written).
2. **Replay.** During incident recovery or new-service onboarding, a service must be able to replay past events from a given offset.
3. **Fan-out.** A single `telemetry.span_enriched` event is consumed by Intelligence (for scoring) and potentially future services.
4. **Tenant isolation.** Events for tenant A must never be processed by tenant B's consumer group.
5. **Audit trail.** Events are part of the audit log — they should be retained for a configurable period.
6. **MSK compatibility.** Existing infra investment: the platform runs on AWS. We want a managed Kafka rather than a self-managed broker.

## Decision

Use **Amazon MSK (Managed Streaming for Apache Kafka)** as the canonical event bus for all inter-service async communication.

All producers and consumers use `@aeos/event-bus-client` (TypeScript) or `aeos_event_bus_client` (Python). Services never construct Kafka topic names or producer/consumer configs manually.

### Topic Naming Convention

```
aeos.{tenant_id}.{domain}.{event_type}
```

| Segment | Description | Example |
|---|---|---|
| `aeos` | Platform namespace — always literal | `aeos` |
| `{tenant_id}` | Tenant UUID — enforced by event-bus-client | `tenant_7f3a2b1c` |
| `{domain}` | Service domain — lowercase | `telemetry`, `ledger`, `registry` |
| `{event_type}` | Event name — snake_case | `span_enriched`, `variance_detected` |

**Full examples:**

```
aeos.tenant_7f3a2b1c.telemetry.span_enriched
aeos.tenant_7f3a2b1c.ledger.row_written
aeos.tenant_7f3a2b1c.ledger.variance_detected
aeos.tenant_7f3a2b1c.registry.uop_mapped
aeos.tenant_7f3a2b1c.recommendations.created
aeos.tenant_7f3a2b1c.governance.attestation_generated
```

Topic names are generated by `@aeos/event-bus-client`. Never hardcode topic strings.

### Consumer Group Naming

```
{service}.{tenant_id}.{domain}.{event_type}
```

Example: `intelligence.tenant_7f3a2b1c.telemetry.span_enriched`

This ensures each service-per-tenant has an independent consumer group and offset position.

## Alternatives Considered

### Option A: AWS SQS + SNS (rejected)

**Rejected.**

SQS/SNS is AWS-native and simple. However:

- **No ordering guarantee** across messages in a standard SQS queue. FIFO queues add ordering but have throughput limits (3,000 messages/sec per queue).
- **No replay.** Once a message is consumed and deleted from SQS, it is gone. We cannot replay past events for a new consumer or during incident recovery.
- **Fan-out complexity.** Fan-out requires an SNS topic + N SQS subscriptions. At scale (many tenants × many event types), managing thousands of SQS queues becomes operationally complex.

### Option B: Amazon EventBridge (rejected)

**Rejected.**

EventBridge is excellent for low-volume routing but:

- **No replay.** EventBridge Archive + Replay exists but is operationally cumbersome for high-throughput scenarios.
- **Throughput limits.** Default: 10,000 events/sec per account. At scale, we approach this ceiling.
- **Cost.** EventBridge charges per event; at high volume, Kafka (fixed MSK cluster cost) is significantly cheaper.
- **Schema registry** integration is weaker than Kafka's native Schema Registry.

### Option C: RabbitMQ (rejected)

**Rejected.**

RabbitMQ is a proven message broker, but:

- **No ordering guarantee at queue level** in most configurations.
- **No replay.** RabbitMQ queues are consumption-destructive (like SQS).
- **Self-managed.** AWS does not offer a managed RabbitMQ comparable to MSK. We'd need to operate RabbitMQ clusters ourselves or use a third-party managed service.
- **Not AWS-native.** Adds operational complexity and cross-service IAM integration is weaker.

### Option D: Kafka (MSK) — selected

Selected because:

- **Ordering guarantees.** Within a partition, events are strictly ordered. We partition by `tenant_id` to ensure all events for a tenant maintain order.
- **Replay.** Configurable retention (default: 7 days). New consumers start from offset 0 to replay all events.
- **Fan-out.** Multiple consumer groups on the same topic — zero overhead for adding a new consumer.
- **MSK = managed.** No cluster ops, AWS-native IAM auth, VPC-native networking.
- **Existing tooling.** `kafkajs` (TS), `confluent-kafka-python` — mature, well-maintained.
- **Tenant isolation via topic naming.** `aeos.{tenant_id}.{domain}.{event_type}` gives us complete tenant isolation with zero application-layer filtering needed at the broker level.

## Consequences

### Positive

- Services are fully decoupled — a Recommendations service outage does not affect Telemetry ingestion.
- New services can subscribe to existing topics without any producer-side changes.
- Replay enables incident recovery and new-service bootstrap.
- `@aeos/event-bus-client` enforces topic naming, schema validation, and tenant isolation — low risk of misconfiguration.

### Negative / Mitigations

| Consequence | Mitigation |
|---|---|
| MSK cluster is an always-on cost | Justified: events per day at scale make MSK cheaper than SQS/EventBridge per-event pricing |
| Kafka requires consumer group management | `@aeos/event-bus-client` handles this automatically |
| Message schema evolution requires care | Use canonical schema types from `@aeos/canonical-schema`; additive changes only — no field removals |
| Local dev needs a Kafka broker | `local-dev/docker-compose.yml` includes KRaft-mode Kafka (no ZooKeeper) |
| On-prem mode: customer must run Kafka | Kafka is Apache 2.0; well-understood ops. Documented in on-prem runbook |
| Dead-letter queue handling | Each consumer implements a DLQ pattern via a `{topic}.dlq` topic; `@aeos/event-bus-client` provides helpers |

## Implementation Notes

- MSK cluster config: `infra/terraform/modules/msk/`.
- Kafka topics created at tenant provision time by `local-dev/seed/kafka-topics.sh` (local) and `services/substrate/src/provisioning/kafka-topics.ts` (production).
- Retention: 7 days default, configurable per topic via Terraform.
- Replication factor: 3 (MSK default). Min in-sync replicas: 2.
- Compression: `lz4` (configured in `@aeos/event-bus-client` producer defaults).
- See [service-map.md](../service-map.md) for the full list of events produced and consumed per service.
